# 8. Requirements

## 8.1 Functional Requirements

| ID | Requirement | Priority | Aggregate/Context |
|----|-------------|----------|-------------------|
| FR-01 | The system shall allow Customer accounts to register using a unique email address and username (3–30 characters) with a BCrypt-hashed password, and to authenticate via `POST /auth/login` to receive a signed JWT. | Must | User |
| FR-02 | The system shall expose a read-only product catalog endpoint accessible to all authenticated roles, returning only products where `isActive = true`. | Must | Product |
| FR-03 | The system shall allow authenticated Customers to initiate a purchase (`POST /sales`) against an active product in a stocked machine slot, triggering payment processing via the external gateway; `UnitPrice` shall be sourced exclusively from the server-side catalog record. | Must | Sale |
| FR-04 | The system shall allow Customers to retrieve their own purchase history (`GET /customers/{id}/history`), scoped strictly to the requesting user's identity. | Must | Sale |
| FR-05 | The system shall allow Operators to view current stock levels and telemetry data for vending machines assigned to their account. | Must | VendingMachine |
| FR-06 | The system shall allow Operators to restock product quantities in assigned machine slots, subject to the slot's defined `SlotCapacity` constraint. | Must | VendingMachine |
| FR-07 | The system shall allow Operators to report machine issues, transitioning machine status to `MAINTENANCE`. | Should | VendingMachine |
| FR-08 | The system shall allow Administrators to create, update, deactivate, and reprice products in the catalog; price changes shall take effect immediately for new sales. | Must | Product |
| FR-09 | The system shall allow Administrators to manage user accounts, including role assignment (`CUSTOMER`, `OPERATOR`, `ADMINISTRATOR`), suspension (`SUSPENDED`), and locking (`LOCKED`). | Must | User |
| FR-10 | The system shall allow Administrators to trigger on-demand MySQL database backups via `POST /admin/backup`; backups shall be AES-256-GCM encrypted and stored in `/var/vendnet/backups/{YYYY-MM-DD}/`. | Must | System |
| FR-11 | The system shall allow Administrators to generate operational reports (sales summaries, stock levels, maintenance logs) on-demand, stored in `/var/vendnet/reports/{type}/{YYYY}/{MM}/{DD}/`. | Must | System |
| FR-12 | The system shall ingest telemetry events (machine status, slot stock levels, sale events) from vending machines over mTLS, persisting them to the database. | Must | VendingMachine |
| FR-13 | The system shall process payment gateway callbacks via `POST /payments/webhook`, verifying HMAC-SHA256 signatures and updating `Sale` records based on validated payment status. | Must | Sale |
| FR-14 | The system shall maintain an immutable, append-only audit log of all authentication events, data mutations, and administrative actions, with per-entry HMAC-SHA256 checksums. | Must | System |
| FR-15 | The system shall enforce Role-Based Access Control with exactly three roles (Customer, Operator, Administrator) as defined in the RBAC Permission Matrix (§1.4); the system shall return HTTP 401 for unauthenticated requests and HTTP 403 for authorised users accessing resources outside their role. | Must | System |
| FR-16 | The system shall push product price updates and slot configuration changes to registered vending machines via mTLS (`ROLE_MACHINE` authenticated connection) following any Administrator-initiated pricing or configuration change. | Should | VendingMachine |

## 8.2 Non-Functional Requirements

| ID | Requirement | Category | Justification |
|----|-------------|----------|---------------|
| NFR-01 | All API endpoint responses shall complete within 500 ms at the 95th percentile under normal operating load (≤ 500 concurrent users). | Performance | End-user experience and SLA baseline for the companion app and Operator tooling. |
| NFR-02 | The system shall achieve 99.9% uptime, excluding planned maintenance windows of up to 4 hours per calendar month. | Availability | Vending machines are operational 24/7; extended downtime blocks all purchases and telemetry ingestion. |
| NFR-03 | The system shall support a minimum of 1000 concurrent authenticated users without violating the NFR-01 latency SLA. | Scalability | Fleet growth and peak-hour purchase volume require elastic capacity beyond the initial deployment. |
| NFR-04 | The system shall achieve an RTO of ≤ 4 hours and an RPO of ≤ 24 hours, supported by daily encrypted backups stored in `/var/vendnet/backups/`. | Recoverability | Regulatory and operational continuity requirements for financial transaction data. |
| NFR-05 | All third-party Maven dependencies shall be scanned for known CVEs on every CI pipeline run using OWASP Dependency-Check; builds containing Critical or High severity CVEs shall fail. | Maintainability | Proactive SCA prevents introduction of known-vulnerable libraries into the production artefact. |
| NFR-06 | The system shall be deployable on any Linux-based Docker host via the provided `docker-compose.yml` configuration without environment-specific code changes. | Portability | Reproducible deployments are required for CI/CD, staging, and on-premises production hosting. |

## 8.3 Security Requirements

> All 71 security requirements (SR-01 to SR-71) are derived from the Critical and High threats identified in §6 and the mitigations defined in §7. Requirements marked with pre-assigned identifiers (SR-01 to SR-08) correspond directly to the Abuse Cases in §5 (AC-01 to AC-08). Requirements SR-09 to SR-71 cover the remaining Critical and High threats ordered by OWASP Risk Score descending.

### 8.3.1 Authentication & Access Control

| ID | Requirement | Justification | Threat Ref | ASVS Ref |
|----|-------------|---------------|------------|----------|
| SR-04 | The system shall validate all JWT signatures using exclusively HS256; tokens specifying `alg:none` or any algorithm other than HS256 shall be rejected with HTTP 401. | JWT algorithm confusion enables token forgery without knowledge of the signing key. | T-28 | V9.1.2 |
| SR-11 | The system shall derive role-based `GrantedAuthority` from the database user record at token-validation time; the `role` claim in the JWT payload shall be discarded. | JWT payload tampering escalates privileges without requiring a database write. | T-06 | V8.3.1 |
| SR-12 | The system shall require TOTP-based multi-factor authentication for all accounts holding the `ADMINISTRATOR` role before issuing an admin-scoped JWT. | Administrator account takeover grants full system control including pricing, user management, and backups. | T-12 | V6.3.3 |
| SR-14 | The `SecurityFilterChain` bean shall include an explicit `requestMatchers` entry for every protected endpoint; `POST /sales/**` shall require the `CUSTOMER` or `ADMINISTRATOR` role. | A missing security filter entry allows unauthenticated callers to submit sale transactions. | T-50 | V8.2.1 |
| SR-21 | The system shall invalidate all existing JWTs for a user immediately on password change by binding token validity to a per-user credential version stored in the database. | A recovered or leaked HMAC secret allows forged tokens to pass validation unless old tokens are invalidated independently of the secret. | T-30 | V9.1.2, V7.4.1 |
| SR-23 | The system shall restrict all pricing write operations (`PUT /products/{id}/price`) exclusively to the `ADMINISTRATOR` role; requests from any other role shall return HTTP 403. | A rogue or lower-privileged actor setting product prices to zero causes direct revenue loss across the entire catalog. | T-65 | V8.2.1 |
| SR-25 | The system shall rate-limit `POST /auth/login` to a maximum of 10 requests per originating IP address per minute, returning HTTP 429 with a `Retry-After` header on breach. | Unrestricted login attempts enable automated credential stuffing leading to account takeover. | T-01 | V6.3.1 |
| SR-26 | The system shall enforce ownership verification on `GET /customers/{id}/history`; requests where the JWT `sub` does not match `{id}` and the caller lacks `ADMINISTRATOR` role shall return HTTP 404. | Horizontal IDOR exposes other customers' purchase histories, violating GDPR data-minimisation obligations. | T-04 | V8.2.2 |
| SR-27 | The system shall log every role-assignment and account-status change with the acting Administrator's user ID, target user ID, previous value, new value, and UTC timestamp. | Rogue administrative changes to roles or account status must be attributable and reversible via an audit trail. | T-13 | V8.3.1, V16.3.1 |
| SR-30 | The system shall generate the JWT HMAC signing key using `Keys.secretKeyFor(SignatureAlgorithm.HS256)` (minimum 256-bit entropy) and shall validate key entropy at application startup. | A weak JWT secret is recoverable by offline brute-force, enabling unlimited token forgery. | T-29 | V9.1.1, V11.5.1 |
| SR-31 | The system shall apply a per-IP Bucket4j rate limit of 10 login requests per minute on `POST /auth/login`; BCrypt work factor shall be configured at 12 to bound per-attempt cost. | BCrypt amplification at higher work factors combined with a high request rate exhausts the Tomcat thread pool. | T-34 | V6.3.1 |
| SR-33 | The system shall add the `jti` claim of any JWT belonging to a `SUSPENDED` or `LOCKED` account to a Redis revocation blocklist; every authenticated request shall check the blocklist before proceeding. | A suspended Administrator's unexpired JWT continues to authorise pricing or user-management operations until it expires. | T-64 | V7.4.2 |
| SR-43 | The system shall require TOTP-based MFA for all `OPERATOR` accounts and shall emit a `NEW_DEVICE_LOGIN` security event on first login from an unrecognised device fingerprint (User-Agent + IP subnet). | Credential theft via spear-phishing bypasses single-factor authentication for Operator accounts with machine access. | T-07 | V6.3.3 |
| SR-44 | The system shall filter all telemetry and machine-log API responses to machines in the requesting Operator's assigned fleet; requests for machines outside that assignment shall return HTTP 404. | Cross-machine IDOR exposes operational telemetry from machines belonging to other fleet segments. | T-10 | V8.2.2 |
| SR-45 | The system shall annotate every Administrator-only service method with `@PreAuthorize("hasRole('ADMINISTRATOR')")` and shall enable `@EnableMethodSecurity` at the application service layer. | A missing `@PreAuthorize` annotation on a service method allows an authenticated Operator to invoke Admin-only functionality. | T-11 | V8.2.1 |
| SR-47 | The system shall maintain a JWT revocation blocklist in Redis keyed on the `jti` claim; suspended accounts' active tokens shall be added to the blocklist at the point of suspension. | An unexpired JWT for a suspended account remains cryptographically valid and can be replayed unless explicitly revoked. | T-35 | V7.4.1 |
| SR-48 | The system shall apply `@PreAuthorize` annotations at the application service layer for all role-restricted operations, in addition to controller-layer security, enforcing defence-in-depth. | A controller-only annotation is bypassed if a service method is called directly from another service without passing through the controller security layer. | T-36 | V8.3.1 |
| SR-49 | The system shall validate Operator-to-machine assignment via a `MachinePermissionEvaluator` before processing any stock-update write operation; the `ADMINISTRATOR` role shall bypass the assignment check. | IDOR on stock-update endpoints allows an Operator to modify inventory on machines outside their assigned fleet. | T-38 | V8.2.2 |
| SR-52 | The system shall restrict `GET /sales/{saleId}` to the owning Customer's JWT `sub` claim; the endpoint shall return HTTP 404 (not HTTP 403) when the sale is not found or is not owned by the requesting user. | IDOR on the sale-retrieval endpoint exposes other Customers' purchase records and transaction references. | T-48 | V8.2.2 |
| SR-55 | The system shall restrict `POST /admin/backup` to the `ADMINISTRATOR` role; requests from any other role shall return HTTP 403. | An RBAC misconfiguration allowing a lower-privileged role to trigger backups enables exfiltration of encrypted backup archives. | T-57 | V8.2.1 |
| SR-58 | The system shall annotate `PricingController.updatePrice()` with `@PreAuthorize("hasRole('ADMINISTRATOR')")` and shall include an integration test asserting that an `OPERATOR`-authenticated request receives HTTP 403. | An overly permissive annotation allows Operators to modify product pricing, violating business rules. | T-69 | V8.2.1 |
| SR-62 | The system shall include a `userId` predicate on every JPA repository query that returns user-scoped records, enforced via JPQL named-parameter binding. | A missing `userId` filter in a JPA query returns all users' records to the first requester, exposing PII in bulk. | T-91 | V8.2.2 |
| SR-63 | The system shall apply ownership predicates to all paginated and list queries on user-scoped resources; cross-user isolation shall be verified by integration tests using distinct test users. | An over-broad JPA query without a scope predicate leaks records belonging to other users in paginated responses. | T-112 | V8.2.2 |
| SR-64 | The system shall restrict the audit log read API (`GET /admin/audit`) to the `ADMINISTRATOR` role; requests from `OPERATOR` or `CUSTOMER` roles shall receive HTTP 403. | Exposing the full security event history to lower-privileged roles reveals infrastructure details and attack patterns. | T-116 | V8.2.1 |

### 8.3.2 Data Security

| ID | Requirement | Justification | Threat Ref | ASVS Ref |
|----|-------------|---------------|------------|----------|
| SR-03 | The system shall derive `UnitPrice` for every sale exclusively from the server-side `Product` catalog record; any `unitPrice` field present in the `SaleRequestDto` request body shall be annotated `@JsonIgnore` and discarded before processing. | Client-supplied price fields enable revenue loss by purchasing products below catalog price. | T-46 | V8.3.1 |
| SR-15 | The system shall emit a `BULK_EXPORT` audit event for any API response returning more than 1000 records in a single call, recording the user ID, endpoint, record count, and UTC timestamp. | A compromised Administrator account can exfiltrate the entire customer and transaction database undetected without bulk-access alerting. | T-15 | V16.3.1 |
| SR-18 | The system shall store all database credentials (URL, username, password) exclusively in OS environment variables referenced by Spring property placeholders; no plaintext credentials shall appear in source code or repository history. | Leaked credentials in source code or configuration files grant direct database access bypassing all application-layer controls. | T-70 | V13.3.1 |
| SR-19 | The system shall configure MySQL with `bind-address=127.0.0.1` to reject external connections; the application service account shall be granted access only to the `vendnet` schema from `127.0.0.1`. | Direct external database access bypasses all application-layer audit controls and allows raw SQL manipulation of financial records. | T-71 | V14.1.1 |
| SR-20 | The application database account shall hold only `SELECT`, `INSERT`, `UPDATE`, `DELETE` on `vendnet.*`; all DDL operations shall be performed by a separate Flyway migration account that is disabled after schema deployment. | Excessive database privileges allow an attacker with application credentials to execute `DROP TABLE` or `ALTER TABLE` operations. | T-77 | V8.2.1 |
| SR-35 | The system shall classify report files containing PII as `RESTRICTED`; report-serving endpoints shall verify that the requesting user holds `ADMINISTRATOR` role before returning file contents. | Report files containing customer PII served to Operator-role users violate data-minimisation and access-control requirements. | T-82 | V14.1.1 |
| SR-37 | The system shall set `Cache-Control: no-store` and `Pragma: no-cache` response headers on all endpoints returning authentication tokens or personal data. | Browser caches retaining JWT responses enable session theft from shared or public terminals. | T-96 | V14.3.2 |
| SR-38 | The system shall annotate `User.passwordHash` with `@JsonProperty(access = JsonProperty.Access.WRITE_ONLY)`; all user-facing API responses shall use a `UserResponseDto` that excludes the `passwordHash` field. | Serialising BCrypt hashes in API responses enables offline brute-force of user credentials. | T-100 | V14.1.1 |
| SR-51 | The system shall record a `transactionRef` (UUID v4, sourced from the payment gateway response) on every `Sale` record; a sale shall not transition to `COMPLETED` status without a non-null `transactionRef`. | Missing transaction references prevent chargeback dispute resolution and break the financial audit trail. | T-47 | V11.5.1 |

| SR-61 | Backup files shall be encrypted using AES-256-GCM envelope encryption; the plaintext MySQL dump shall be deleted immediately after successful encryption; the AES key shall be stored separately from the backup archive. | An OS-level attacker with read access to the backup directory can exfiltrate the entire database if backups are stored unencrypted. | T-78 | V11.3.2 |

### 8.3.3 Communication Security

| ID | Requirement | Justification | Threat Ref | ASVS Ref |
|----|-------------|---------------|------------|----------|
| SR-08 | The system shall apply a fleet-level aggregate rate limit of 100 telemetry write operations per second across all vending machines; limit breaches shall return HTTP 429 and emit a `FLEET_RATE_LIMIT_EXCEEDED` audit event. | A coordinated telemetry flood from the full machine fleet saturates the database write path and causes availability loss. | T-55 | V2.4.1 |
| SR-36 | The system shall apply a per-authenticated-user rate limit of 100 requests per minute on all endpoints, keyed on the JWT `sub` claim, returning HTTP 429 with a `Retry-After` header on breach. | High-frequency authenticated requests from a single JWT exhaust Tomcat thread pool resources, causing availability degradation for all users. | T-88 | V2.4.1 |
| SR-40 | The system shall implement an outbound HTTP allowlist for all Payment Gateway calls; requests targeting hostnames or IP ranges not on the allowlist shall be rejected before transmission. | SSRF via Payment Gateway integration allows an attacker to redirect outbound payment requests to internal services or external attacker infrastructure. | T-106 | V1.3.6, V13.2.4 |
| SR-41 | The system shall enforce TLS 1.3 on the payment callback listener, restricting cipher suites to `TLS_AES_256_GCM_SHA384` and `TLS_CHACHA20_POLY1305_SHA256`. | In-transit modification of payment callback payloads enables FAILED-to-COMPLETED status manipulation and fraudulent order fulfilment. | T-109 | V12.1.1 |
| SR-46 | The system shall apply a per-machine rate limit of 2 telemetry write operations per minute, keyed on the mTLS client certificate CN, returning HTTP 429 with `Retry-After: 60` on breach. | A single compromised machine can flood the telemetry endpoint, exhausting database write capacity and masking legitimate machine status. | T-22 | V2.4.1 |
| SR-53 | The system shall apply a per-user rate limit of 10 purchase requests per minute on `POST /sales`, keyed on the JWT `sub` claim, returning HTTP 429 on breach. | Unrestricted purchase requests from a single user exhaust the Payment Gateway API quota, blocking legitimate purchases network-wide. | T-49 | V2.4.1 |
| SR-65 | The system shall require mTLS client certificate authentication for all machine-to-server communication; certificate CN binding shall restrict each machine to its registered identity. | A machine without mTLS can send falsified sale events with arbitrary quantities, corrupting financial records. | T-19 | V12.1.3 |
| SR-67 | The system shall apply fleet-level rate limiting across all machine telemetry sources using a shared Redis counter; aggregate ingestion exceeding 100 events per second shall trigger a Resilience4j circuit breaker and emit a `FLEET_FLOOD_DETECTED` audit event. | Coordinated telemetry floods from multiple compromised machines simultaneously saturate the ingestion pipeline in a manner that per-machine limits cannot prevent individually. | T-103 | V2.4.1 |
| SR-68 | The system shall verify mTLS client certificates against CRL/OCSP on every connection; certificates with revoked status shall be rejected with a TLS `certificate_revoked` alert. | A stolen mTLS client certificate from a decommissioned machine can be replayed until it expires if revocation is not checked in real time. | T-18 | V12.1.3, V12.1.4 |
| SR-69 | The system shall scope machine-issued mTLS certificates (`ROLE_MACHINE`) to specific API paths (`/telemetry/**`, `/machines/{id}/sales`) using `SecurityFilterChain` `requestMatchers`; attempts to access admin or user paths shall return HTTP 403. | A machine certificate used to call admin or customer endpoints bypasses human-identity authorization controls. | T-23 | V8.2.2, V12.1.3 |
| SR-70 | The system shall validate the originating IP address of mTLS connections against the machine's registered static IP or CIDR range; connections from unexpected IP ranges shall trigger a `CERT_IP_MISMATCH` security event and be denied. | A stolen mTLS certificate used from attacker-controlled cloud infrastructure is detected by IP-range enforcement even if the certificate itself has not yet been revoked. | T-101 | V12.1.4 |
| SR-71 | The system shall configure `server.ssl.client-auth=need` on all endpoints accessed by vending machines; machines lacking a valid mTLS client certificate shall be refused at the TLS handshake level. | Legacy machines without mTLS can receive unauthorised configuration pushes that modify firmware parameters. | T-104 | V12.1.3 |

### 8.3.5 Third-Party Components

| ID | Requirement | Justification | Threat Ref | ASVS Ref |
|----|-------------|---------------|------------|----------|
| SR-02 | The system shall validate every inbound payment webhook by computing HMAC-SHA256 over the raw request body before JSON deserialisation and comparing the result in constant time using `MessageDigest.isEqual()`; requests failing verification shall return HTTP 400 without processing. | A forged payment webhook bypassing HMAC verification allows an attacker to mark unpaid transactions as completed, causing direct financial loss. | T-108 | V9.1.1 |

### 8.3.6 Logging & Monitoring

| ID | Requirement | Justification | Threat Ref | ASVS Ref |
