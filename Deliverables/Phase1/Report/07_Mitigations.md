# 7. Mitigations

## 7.1 Methodology

Security mitigations for VendNet are designed around the principle of **defense-in-depth**: no single control is assumed to be sufficient, and each attack vector identified in §4 is addressed by at least one primary control and, where the risk level warrants it, one or more secondary controls operating at a different layer of the stack. The prioritisation order follows directly from the OWASP Risk Rating scores in §6: all 22 Critical-rated threats are addressed before any High-rated threat, and mitigations are ordered by descending Risk Score within each band. This ensures that the highest-probability, highest-impact scenarios — SQL injection enabling full database extraction, JWT algorithm bypass granting arbitrary administrative identity, `ProcessBuilder` command injection enabling OS-level compromise, and forged payment webhooks enabling free product dispensing — receive engineering attention in Phase 2 before lower-severity concerns are addressed. The eight abuse cases (AC-01 to AC-08) directly motivated the first eight mitigation identifiers (M-01 to M-08) and their corresponding security requirements (SR-01 to SR-08); the mitigations for those threats are specified in sufficient detail to be directly implementable by the Phase 2 development team.

All mitigations are grounded in OWASP guidance: OWASP Top 10 (A01–A10, 2021 edition), OWASP Application Security Verification Standard (ASVS) 5.0, and OWASP API Security Top 10 (2023 edition). Where ASVS 5.0 specifies a testable verification criterion (e.g., V3.5.3 for JWT algorithm enforcement, V5.3.4 for parameterised queries, V14.4.5 for security response headers), the Implementation Notes column identifies the corresponding Spring Boot or Java 17 mechanism that satisfies it. Implementation Notes are technology-specific: they name actual Spring Security annotations, `application.properties` configuration keys, JJWT API calls, Hibernate `@Query` patterns, MySQL privilege grant statements, or systemd unit directives so that Phase 2 developers can translate each row directly into a pull request without ambiguity.

## 7.3 Defense-in-Depth Strategy

**Perimeter Layer.** The outermost layer of VendNet's security architecture enforces transport-level guarantees and coarse-grained request filtering before any application logic executes. All external connections — Customer REST API (`DF1/DF2`), Operator management flows (`DF3/DF4`), Administrator management commands (`DF5/DF6`), Vending Machine telemetry and sales channels (`DF7/DF8`, `EP2`), and the Payment Gateway callback endpoint (`EP3`) — must use TLS 1.3 with a restricted cipher suite (`TLS_AES_256_GCM_SHA384`, `TLS_CHACHA20_POLY1305_SHA256`) configured via `server.ssl.enabled-protocols=TLSv1.3` in the embedded Tomcat connector. The machine-facing channel additionally requires mutual TLS (`server.ssl.client-auth=need`) backed by CRL/OCSP certificate revocation checking (SR-68 to SR-70). Rate limiting is the second perimeter control: per-IP Bucket4j filters on `POST /auth/login` (SR-25, SR-31) and per-user Resilience4j rate limiters on `POST /sales` (SR-53) and bulk Admin endpoints (SR-15, SR-36) prevent credential-stuffing floods, purchase-quota exhaustion, and mass-exfiltration scripting before they reach the application layer. Per-machine rate limiters on the telemetry endpoint (SR-08, SR-46, SR-67) prevent a compromised VM fleet from saturating the shared MySQL instance. Input validation at the perimeter — Bean Validation `@Size`, `@Min`/`@Max`, and `@Pattern` annotations on all request DTOs combined with `spring.jackson.deserialization.fail-on-unknown-properties=true` and Tomcat's `max-http-form-post-size` limit — rejects malformed or oversized payloads before deserialisation, reducing the attack surface for deserialisation exploits (SR-32, SR-50).

**Application Layer.** The application layer enforces identity, authorisation, and business-rule integrity for every request that passes the perimeter. JWT security is hardened at three points: algorithm enforcement via `SignatureAlgorithm.HS256` in the JJWT parser (SR-04) eliminates the `alg:none` bypass; a 256-bit random signing key generated with `Keys.secretKeyFor(HS256)` (SR-30) raises the brute-force bar; and a custom `JwtAuthenticationConverter` that loads the authoritative `GrantedAuthority` from `userRepository` rather than trusting the JWT `role` claim (SR-11, SR-21) neutralises the payload-tampering family of attacks (T-06, T-30). The JWT blocklist in Redis (SR-47, SR-33) ensures suspension is effective immediately, closing the stateless-token revocation gap. Spring Security `@PreAuthorize` annotations are enforced at both the controller layer and, via `@EnableMethodSecurity`, at the service layer (SR-45, SR-48) so that internal callers such as scheduled tasks and event listeners cannot bypass role checks. The `@PreAuthorize("hasRole('MACHINE')")` scoping on telemetry endpoints and its absence on human-role endpoints (SR-69) prevents machine certificates from calling administrative paths. Business-rule integrity controls include server-side price resolution (`@JsonIgnore` on `unitPrice`, SR-03), pessimistic-write locking on the `Slot` row during purchase processing to prevent TOCTOU overselling (SR-07), and plausibility checks on firmware-submitted sale quantities validated against the last telemetry record (SR-39, SR-65). Payment webhook forgery is addressed by raw-body HMAC-SHA256 validation with constant-time comparison (SR-02, SR-13, SR-28) and idempotency enforcement via a `processed_webhooks` unique constraint. `ProcessBuilder` command injection is eliminated by replacing all user-supplied input in OS-operation endpoints with a closed `enum BackupType` and a fully hardcoded argument array (SR-01, SR-16), combined with server-side-only filename generation (SR-01).

**Data Layer.** The data layer ensures that even if perimeter and application controls are bypassed, the impact of a compromise is limited by data-level controls on the MySQL instance, the `/var/vendnet/` file system, and the cryptographic integrity of stored artefacts. MySQL is protected by three complementary controls: the application DB account holds only `SELECT, INSERT, UPDATE, DELETE` with no DDL privileges (`GRANT` statement in SR-19 and SR-20), and Flyway schema changes run under a separate migration account; the `max_execution_time=5000ms` and HikariCP connection timeout (SR-60) abort slow injected queries before they exhaust the connection pool; and the binary log in `ROW` format with a 90-day retention policy (SR-19, SR-59) provides an authoritative, tamper-evident record of all direct-access operations that bypass the application audit log. File-system controls on `/var/vendnet/` operate at three sub-layers: directory permissions (`700` on `/var/vendnet/backups/`, `640` on log files) backed by a least-privilege `vendnet` service account confined via systemd's `NoNewPrivileges=true` and `ReadWritePaths` directives (SR-17, SR-56, SR-61) prevent OS-level lateral movement; path traversal is blocked by the normalise-then-prefix-assert pattern applied in both the directory-creation path and the file-write path (SR-06, SR-22, SR-24, SR-66); and backup archives are protected by AES-256-GCM envelope encryption with a DEK encrypted by a Vault-managed master key, ensuring that file-readable backup files yield no plaintext data without access to the secrets manager (SR-56, SR-61). Audit-log integrity is maintained by appending an HMAC-SHA256 checksum to every log entry using a key stored separately from the application signing key, with a nightly verification job that triggers a Spring Boot Actuator health failure on any tampered entry (SR-34). Together these data-layer controls bound the blast radius of a full application compromise to encrypted, HMAC-verified artefacts that cannot be silently modified without detection.
